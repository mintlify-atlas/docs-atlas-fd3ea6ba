WPX ships with pre-built plugin lists that tell it which slugs to probe during a scan. These lists are ranked by popularity so the most impactful plugins are always checked first.

## Files

```
data/
├── plugins_active.txt
├── plugins_dead.txt
├── plugins_catalog.json
└── plugins_dead.jsonl
```

### `data/plugins_active.txt`

A ranked list of active plugin slugs, one per line. Plugins are ordered by **popularity score** — the geometric mean of active install count and total download count. The default list contains the top 5,000 plugins.

This is the primary source WPX draws from when scanning.

### `data/plugins_dead.txt`

A ranked list of closed and removed plugin slugs, ordered by historical install count. The default list contains the top 2,500 dead plugins.

Dead plugins are still worth scanning: many sites continue to run a plugin after it has been pulled from the WordPress.org directory, and those unmaintained installs are often vulnerable.

### `data/plugins_catalog.json`

A JSON file that caches metadata for every active plugin fetched from the WordPress.org API. Each entry maps a plugin slug to its install count, download count, star rating, and update dates. The plugin fetcher writes and reads this file as a checkpoint during catalog builds.

### `data/plugins_dead.jsonl`

An append-only JSONL cache of metadata for closed plugins. Each line is a JSON object. When WPX loads this file, the last entry for any given slug wins (last-write-wins), so new enrichment data from Archive.org automatically overrides older stale records without needing to rewrite the whole file.

## Plugin source selection

When a scan starts, WPX checks for a file named `plugins_full.txt` in three locations, in order:

1. `data/plugins_full.txt`
2. `plugins_full.txt`
3. `.wpx_data/plugins_full.txt`

If none of those files exist, WPX falls back to the WPScan default list at `.wpx_data/plugins.txt` (~1,500 slugs downloaded from `data.wpscan.org`).

<Note>
  `plugins_full.txt` is not shipped with WPX. It is meant to be generated by the plugin fetcher when you want a single merged list for `--full-scan`. If you only use the default scan depth, you do not need this file.
</Note>

## Scan depth

WPX does not scan every slug in the list by default. It scans the **top 200 plugins** unless you override the limit.

| Flag | Behavior |
|------|----------|
| _(default)_ | Scans the top 200 slugs from the active list |
| `--plugins-limit N` | Scans the top N slugs |
| `--full-scan` | Scans every slug in the list (50,000+) |

```bash
# Default: top 200
python3 wpx.py -u https://example.com

# Top 1,000 plugins
python3 wpx.py -u https://example.com --plugins-limit 1000

# Every slug ever (can take several hours)
python3 wpx.py -u https://example.com --full-scan
```

<Warning>
  `--full-scan` performs tens of thousands of requests. It can take several hours depending on thread count and the target's responsiveness.
</Warning>

## Checking and updating the catalog

To see how old your catalog is, check the file modification time:

```bash
ls -lh data/plugins_active.txt data/plugins_dead.txt
```

To rebuild the catalog from the latest WordPress.org data:

```bash
python3 data/wpx_fetch_plugins.py --sort-by score
```

This updates both `data/plugins_active.txt` and `data/plugins_dead.txt`. See [Plugin fetcher](/data/plugin-fetcher) for the full list of options.
